# Финразбор кредитной истории

Веб-система для менеджера по продажам БФЛ / РДГ. Она превращает кредитный
отчет в понятную финансовую картину: показывает, сколько клиент уже отдал
банкам, куда ушли платежи, сколько еще предстоит выплатить и как эти суммы
соотносятся со стоимостью процедуры.

**Цель продукта** — повысить конверсию консультации и качество квалификации,
не смешивая продажные расчеты с рисками, которые должен оценивать юрист.

Рабочее демо: **<https://test1.baggrisha.xyz>**

## Возможности

- загрузка любого количества PDF или ZIP кликом или перетаскиванием и переключение между отчетами;
- сверка количества договоров и текущего долга со сводкой ОКБ;
- разбор действующих и закрытых договоров, сумм выдачи, долга, ставок/ПСК,
  платежей в тело, процентов и иных начислений;
- расчет оставшегося срока, полной суммы выплат банкам и переплаты;
- редактируемое сравнение «платить банкам / БФЛ / РДГ» с экономией, сроком и
  ежемесячной нагрузкой;
- отдельная проверка договоров с 0–2 платежами, займов, оформленных менее чем
  за 4 дня, и крупных долгов с небольшим числом платежей;
- детализация каждого договора, светлая и темная темы, адаптивная и печатная версии;
- копирование аргумента для клиента и структурированной сводки для юриста;
- визуальная оценка ситуации по долгу и количеству рисковых сигналов.

Параметры по умолчанию: БФЛ — `200 000 ₽ / 15 мес.`, РДГ —
`150 000 ₽ / 15 мес.`. Все параметры можно изменить в интерфейсе; итоговые
цифры пересчитываются, а не подставляются из макета.

## Стек и архитектура

- **Frontend:** React 19, TypeScript, Vite.
- **Backend:** FastAPI, Python 3.12, Pydantic.
- **PDF:** `pdftotext` из Poppler; обработка текстового слоя без OCR.
- **Поставка:** один multi-stage Docker-образ. FastAPI отдает API и собранный
  frontend, поэтому для запуска нужен один контейнер.

Основной поток: браузер извлекает PDF из выбранных ZIP и отправляет каждый PDF в API, backend извлекает текст и
нормализует договоры, выполняет сверку и расчеты, после чего frontend строит
продажный разбор и независимый блок проверки рисков.

## Быстрый запуск

Требования: Docker 24+ и Docker Compose.

```bash
cp .env.example .env
docker compose up --build
```

Приложение будет доступно на <http://localhost:8000>. Для закрытого демо можно
задать `APP_ACCESS_CODE` в `.env`; тот же код потребуется на экране загрузки.

### Запуск для разработки

Backend требует Python 3.12 и системный пакет `poppler-utils`.

```bash
python3.12 -m venv .venv
.venv/bin/pip install -r backend/requirements-dev.txt
PYTHONPATH=backend .venv/bin/uvicorn app.main:app --reload
```

В другом терминале:

```bash
cd frontend
npm install
npm run dev
```

## Расчеты и допущения

Продажные агрегаты включают только действующие договоры. Закрытые договоры
остаются в деталях и участвуют в исторической проверке рисков.

Суммы «уже внесено», «в тело», «проценты» и «иное» берутся из соответствующих
полей ОКБ. Для кредитных карт система не утверждает, что текущий долг уменьшился
на все погашенное тело: после платежа клиент мог повторно использовать лимит.

Источники прогноза выплат банкам применяются по приоритету:

1. Указанная в отчете сумма всех оставшихся платежей.
2. Расчет по текущему долгу, платежу и номинальной ставке.
3. ПСК как приближение эффективной ставки с предупреждением о низкой точности.
4. Ручные значения менеджера, если графика в отчете недостаточно.

Если платеж не покрывает расчетные проценты или данных недостаточно, система не
придумывает срок, а предлагает уточнить платеж и срок. Экономия — это оценка
полной выплаты банкам за вычетом стоимости процедуры. Ежемесячный платеж по
процедуре равен ее стоимости, деленной на срок.

Цвет карточки текущего долга зависит от доли остатка к первоначальной сумме
действующих договоров: `80–100%` — красный, `50–79%` — желтый, менее `50%` —
зеленый. Если исходные суммы указаны не для всех договоров, цвет нейтральный.

Для проверки рисков учитываются уникальные даты положительных операций:
несколько записей в один день считаются одним платежным событием. Найденные
сигналы не являются основанием для отказа или юридическим заключением.

## Поддержка форматов и ограничения

- Поддержаны текстовые отчеты ОКБ `v3.10` и `v3.17`, включая предоставленные
  для задания образцы.
- Формат НБКИ распознается, но без проверочного образца не парсится.
- Сканированные PDF без текстового слоя не поддерживаются: OCR не реализован.
- Нет учетных записей, CRM-интеграции, истории консультаций и постоянного
  хранения результатов. Код доступа предназначен только для закрытого демо.

## Приватность

PDF обрабатывается в памяти процесса и не сохраняется в базе данных или на
диске. Имя файла и содержимое не пишутся в access log. Реальные кредитные
отчеты содержат персональные данные и исключены из Git правилом `*.pdf`.

Восемь полностью вымышленных отчетов с разным долгом, платежной дисциплиной и
рисковыми сочетаниями создаются локально командой:

```bash
node scripts/generate_fake_reports.mjs
```

По умолчанию файлы появятся в
`~/Downloads/Тестовые кредитные отчеты`. Путь можно передать первым аргументом:

```bash
node scripts/generate_fake_reports.mjs /tmp/test-reports
```

## API

`POST /api/v1/reports/analyze` принимает `multipart/form-data` с полем `file`.
Максимальный размер PDF или ZIP — 50 МБ. Суммарный размер распакованных PDF в одном ZIP также ограничен 50 МБ. Если настроен общий код, он передается в
заголовке `X-App-Code`. `GET /api/health` используется для проверки готовности.

Ответ содержит `report`, `summary`, нормализованный массив `contracts`,
`compliance` и `warnings`. Для каждого договора возвращается отдельный прогноз
с источником ставки и уровнем доверия.

## Проверки

```bash
PYTHONPATH=backend .venv/bin/pytest -q backend/tests
cd frontend && npm run build
```

Диагностика отдельного отчета без запуска сервера:

```bash
PYTHONPATH=backend .venv/bin/python backend/scripts/inspect_report.py report.pdf
```
